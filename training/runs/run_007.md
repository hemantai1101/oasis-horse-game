## Why These Features

### Problem diagnosed after Run 006 in-game test

The AI moves toward the axis but fails to:
1. Execute its own 2–3 move win setup
2. Preemptively block opponent 2–3 move wins

**Root cause:** MLP conjunction problem + sparse threat data.

The win condition requires three conditions to hold simultaneously:
- Horse on axis (`on_axis=1`)
- Clear path (`path_clear=1`)
- Friendly backstop occupied (`backstop=+1`)

MLPs struggle to learn 3-way conjunctions from binary features. Additionally, the minimax
teacher always blocks obvious threats, so few examples of "unblocked threat" appear in training data —
the model rarely sees what a real threat looks like.

### Fix: explicit threat count

Instead of hoping the model learns the conjunction internally, we compute it explicitly:
`my_winning_threats` = count of horses where ALL three conditions hold simultaneously.
This gives the model a single number it can directly weight — no conjunction needed.

### Fix: graded path_threat

Binary `path_clear` gives zero signal when the path has 1–2 blockers that could be removed.
Graded `path_threat = max(0, (5 - blocks) / 5.0)` means:
- 1 blocker → 0.8 (almost there)
- 2 blockers → 0.6 (getting closer)
- 5+ blockers → 0.0 (hopelessly blocked)

This creates gradients that reward clearing the path even before the threat is complete.

## New Feature Layout (110 total)

**Per horse × 20 horses = 100 features (unchanged structure):**

| Index offset | Feature | Description |
|-------------|---------|-------------|
| +0 | `col_norm` | (col - 1) / 10 — unchanged |
| +1 | `row_norm` | (row - 1) / 10 — unchanged |
| +2 | `dist_to_center` | Manhattan distance to (6,6) / 10 — unchanged |
| +3 | `on_axis` | 1.0 if col=6 or row=6 — unchanged |
| +4 | `path_threat` | max(0, (5-blocks)/5.0) if on_axis, else 0.0 — **changed** (was binary path_clear) |

**Global × 10 features:**

| Index | Feature | Description |
|-------|---------|-------------|
| 100 | `backstop_(6,5)` | +1=current player, -1=opponent, 0=empty — unchanged |
| 101 | `backstop_(6,7)` | same — unchanged |
| 102 | `backstop_(5,6)` | same — unchanged |
| 103 | `backstop_(7,6)` | same — unchanged |
| 104 | `player_indicator` | 0.0=P1, 1.0=P2 — unchanged |
| 105 | `my_winning_threats / 10` | horses where on_axis + clear path + my backstop set — **new** |
| 106 | `opp_winning_threats / 10` | same for opponent — **new** |
| 107 | `my_horses_at_home / 10` | my pieces in starting corners — **new** |
| 108 | `opp_horses_at_home / 10` | opponent pieces in starting corners — **new** |
| 109 | `my_pieces_blocking_opp / 10` | my axis pieces blocking opp further back — **new** |

**Winning threat logic** (`my_winning_threats`):
- Horse must be on axis (col=6 or row=6), not at center
- Path from horse to center must be fully clear (0 blockers)
- Correct backstop cell for this approach must be occupied by the same player:
  - Horse col=6, row<6 → backstop (6,7)
  - Horse col=6, row>6 → backstop (6,5)
  - Horse row=6, col<6 → backstop (7,6)
  - Horse row=6, col>6 → backstop (5,6)

**Home corners:**
- P1: TL (col≤3, row≤3) + BR (col≥9, row≥9)
- P2: TR (col≥9, row≤3) + BL (col≤3, row≥9)

**Augmentation note:** All 5 new global features [105-109] are invariant under 180° rotation.
`new_feats = list(feats)` already copies them correctly — no augmentation logic change needed.
