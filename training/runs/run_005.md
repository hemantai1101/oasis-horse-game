# Run 005 — 2026-03-15

**Goal:** Fix model capacity bottleneck from Run 004 — wider architecture, same depth=4 data.

**What changed vs Run 004:**

| What | Run 004 | Run 005 | Why |
|------|---------|---------|-----|
| Architecture | 41→128→64→1 | 41→256→128→1 | Run 004 plateaued at 0.707 — model too small to fit depth-4 complexity |
| Params | 14,081 | 44,545 | 3× more capacity |
| Data | same | same | Data not the bottleneck |
| Epochs / LR | same | same | No change |

**Diagnosis from Run 004:** Depth=4 minimax encodes 4-ply lookahead into labels. The 14k-param model
converged at 0.707 with a hard floor and no overfitting — clearly capacity-limited.
Widening the hidden layers (128→256, 64→128) gives more expressive power.

## Notes

- ✅ Wider model confirmed — val loss dropped 0.707 → 0.691 (↓0.016), architecture change was right
- ✅ No overfitting — train/val gap 0.00789, stable throughout
- ⚠️ Gain modest: 3× more params only bought 0.016 improvement — diminishing returns on model size
- ⚠️ Still 0.083 above depth-3 runs (0.691 vs 0.608) — raw position features are the bottleneck
- **Root cause identified:** 41 raw (col, row) features don't encode game-critical concepts:
  - Whether a horse is on the center axis (col=6 or row=6)
  - Whether the path from a horse to center is clear
  - Whether backstop cells (6,5), (6,7), (5,6), (7,6) are occupied and by whom
  - The attacker+backstop setup that is required to win
- More model size or epochs will not fix this — need richer features
- Next lever: **feature engineering** — expand from 41 → 105 game-aware features (Run 006)
