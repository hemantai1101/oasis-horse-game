# Run 013 — 2026-03-29

**Goal:** Fix the AI's passive/defensive play at the architectural level.
Run 012 was stopped early — it only fixed calibration (the 9996 score inflation), not the
root cause of why the AI never attacks.  Run 013 fixes the actual problem: the teacher AI
was trapped in a Nash equilibrium of mutual defensiveness, generating training data that
contained almost no fork-building sequences.

Three coordinated changes break that equilibrium.

---

## Root Cause (why the AI didn't attack)

1. **Defensive Nash equilibrium in self-play** — both players valued blocking (+4,000) more
   than attacking (+3,000), so neither ever committed to building a fork.  Games were long
   positional grinds with no fork attempts.

2. **Fork horizon problem** — building a fork takes 4-5 setup moves.  The heuristic gave
   small per-move bonuses (+400/step) that were always outweighed by the immediate
   defensive payoff.  Minimax chose defense at every ply.

3. **Epsilon exploration never reached attack lines** — top-3 epsilon always picked from
   the defensive heuristic's best moves.  Fork-building sequences (low heuristic score,
   high outcome value) were never explored and therefore absent from training data.

4. **NN features lacked fork-potential signal** — features described the current board but
   not whether you were 1-2 moves away from a fork.  The NN could only learn to value
   positions where a fork was already assembled — but those were always blocked first.

## Change 1 — Fork Bonus in Heuristic (`minimax.py` + `ai-worker.js`)

A fork (2+ simultaneous unblockable threats) is a game-deciding position: the opponent
can only block one.  The +15,000 bonus is deliberately larger than all other heuristic
terms combined, so the teacher minimax will commit to building forks even when individual
steps look unremarkable.

The near-fork bonus (+5,000) rewards positions where the player has one active threat and
one latent threat (on-axis, clear path, but backstop not yet placed) — one piece move
away from a full fork.

Both bonuses are mirrored defensively (-15,000 / -5,000) so both players fight hard to
build and prevent forks, creating genuinely dynamic attacking games.

```
my_active >= 2 (fork):                   +15,000
my_active >= 1 and my_latent >= 1 (near-fork): +5,000
opp_active >= 2:                         -15,000
opp_active >= 1 and opp_latent >= 1:     -5,000
```

A "latent threat" is a horse on-axis with a clear path to center whose required backstop
cell is not yet occupied by the player — one placement away from becoming an active threat.

## Change 2 — 5 New NN Features (`game.py` → `ai-worker.js`)

Feature vector grows from 110 → 115.

| Index | Feature | Description |
|-------|---------|-------------|
| 110 | my_latent_threats / 4 | My horses on-axis + clear path, but no backstop yet |
| 111 | opp_latent_threats / 4 | Same for opponent |
| 112 | my_fork_flag | 1.0 if I have 2+ simultaneous active threats |
| 113 | opp_fork_flag | 1.0 if opponent has 2+ simultaneous active threats |
| 114 | my_near_fork_flag | 1.0 if I have 1 active + 1 latent (one move from fork) |

All 5 new features are invariant under 180° rotation (on-axis property and path-clear are
preserved by 180° rotation; counts and binary flags are therefore also invariant).
No changes needed to the augmentation code in `generate_games.py`.

`train.py` now auto-detects input_size from the first record in the data file, so it
handles 110-feature (old) and 115-feature (new) datasets without manual updates.

## Change 3 — Attack-Biased Epsilon (`minimax.py`)

Half of epsilon explorations (7.5% of all moves) now specifically pick from moves that
advance toward a fork: moves to backstop cells (6,5), (6,7), (5,6), (7,6), or moves that
land a piece on an axis (col=6 or row=6).  If no such move exists, falls back to standard
top-3.

This injects fork-building sequences into training data even in positions where the
heuristic alone wouldn't prioritize them yet.

## What to Check in Browser Testing

Beyond the standard d3/d4 swing and score inflation checks:

1. **Does the AI attempt to build forks?** — Watch if it positions 2 horses on perpendicular
   axes while also placing backstops.  This is the primary success criterion.

2. **d3/d4 swing** — Target < ±1,528 (Run 010 baseline).  Fork positions should score
   dramatically higher at depth=4 than depth=3, which is expected and correct.

3. **Score inflation** — Scores should NOT lock to 9996 early.  Fork bonus (+15,000) will
   push scores higher in genuinely winning positions, but mid-game scores should stay
   varied.

4. **Game length** — Fork-seeking play should shorten games (decisive play vs. positional
   grinding).  Expect games to be shorter than Run 011.

## Run 014 Preview

If Run 013 confirms the teacher AI now builds forks and the NN learns to value them,
Run 014 transitions to iterative NN-as-policy self-play:

- Remove the minimax teacher entirely
- Use the Run 013 NN directly to select moves (no imitation learning)
- Train on actual game outcomes (win/loss signal from real games, not teacher data)
- Repeat for 3-4 iterations

This is the AlphaGo insight applied here: the NN learns from its own wins, not from
imitating a teacher.  Attacking strategies emerge naturally because attacking wins.
