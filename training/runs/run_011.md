# Run 011 — 2026-03-23

**Goal:** Scale up depth=5 training data from 15k (Run 010 validation) to 50k games.
Run 010 proved depth=5 teacher data is far superior to depth=4, but the model was
still declining at epoch 100 — more data + more epochs should push the ceiling lower.
Also adds the repetition-detection fix in `ai-worker.js` to resolve the back-and-forth
stuck behaviour observed during Run 010 browser testing.

---

## Analysis

Val loss converged at **0.641**, higher than Run 010's 0.569. This is not a direct
regression — the two numbers measure different distributions and are not comparable:

| Run | Records | Val set size | Best val loss |
|-----|---------|-------------|--------------|
| Run 010 | 2.4M | ~240k | 0.56910 |
| Run 011 | 10.3M | ~1.03M | 0.64144 |

Run 011's val set is 4× larger and covers far more diverse game positions. Harder
distribution = higher loss even if the model has learned more.

**Root cause of higher loss:** The 3s time limit produced longer games (239 vs 204
records/game). In longer games, early and mid-game positions are nearly 50/50 — both
players could still win from there. These positions have noisy labels, making them
inherently harder to fit regardless of model capacity or training duration.

Train/val gap is only 0.00105 — essentially zero overfitting. The model is fully
converged and has learned everything the dataset can teach it.

**In-game result: regression vs Run 010.** Despite more data and a higher time limit,
Run 011 plays noticeably worse. Two clear symptoms from browser console logs:

---

## Browser Test Results

### Symptom 1 — d3/d4 score swing returned

Run 010 reduced the depth=3 vs depth=4 sign flip to ±1,528. Run 011 brought it back:

```
depth=3 best=9269   depth=4 best=2949   → swing of 6,320
depth=3 best=7886   depth=4 best=1887   → swing of 6,000
depth=3 best=9911   depth=4 best=4611   → swing of 5,300
```

The horizon effect is back — the NN overestimates positions at depth=3 leaves again.

### Symptom 2 — Late-game score inflation locks at 9996

After ~20 moves the AI assigns ~9996 to almost every position:

```
depth=3 best=9996   depth=4 best=9924
depth=3 best=9996   depth=4 best=9898
depth=3 best=9996   depth=4 best=9838
```

When everything scores near the maximum, the AI cannot discriminate between moves.
It makes legal moves but fails to convert obvious winning positions.

### Root cause

The 3s time limit produced longer games (239 records/game vs 204). In longer games,
early and mid-game positions are genuinely ambiguous — both players could still win.
Those positions have noisy labels (+1 or -1 based on a distant outcome) which teaches
the model to be overconfident rather than calibrated. More data amplified this noise.

### Repetition fix — confirmed working ✅

Different pieces were moved each turn throughout the test. The back-and-forth bug
from Run 010 is resolved.

---

## Verdict

❌ **Run 011 is worse than Run 010 in-game.** More data with longer games hurt
label quality. Revert to Run 010 weights. The lesson: for this game, shorter
decisive games produce better-calibrated leaf evaluations than longer complex games.
